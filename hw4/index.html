Overview

In this homework assignment, we implemented a real-time simulation of a cloth. This was done in the following steps: (1) a  system of point masses and springs, heavily relying on PointMasses (2) integrating the physical equations of motion to model how the cloth falls, including equations to simulate the effect of forces (3) implementation for handling collisions with other objects in the scene such as spheres and planes, (4) handling self-collisions, to ensure that the cloth interacts with itself more naturally and (5) shaders, to give the cloth texture, variety, and personality. Overall, this project allowed us to take advantage of various techniques used in the graphics pipeline that allows us to take advantage of the GPU and provide us with valuable insights into various techniques used in not only cloth simulations but also real-time rendering.

Part 1: Masses and springs

  1. Take some screenshots of scene/pinned2.json from a viewing angle where you can clearly see the cloth wireframe to show the structure of your point masses and springs.

For our implementation of cloth simulation, we divided up the cloth into evenly spaced point masses, which are then connected to their neighboring point masses through different types of springs. To accomplish this, we first create a grid with num_width_points by num_height_points of point masses that span the area given by the width and height length parameters. To evenly distribute the masses, we calculate the increments along each dimension, which are width/num_width_points and height/num_height_points. If the cloth's orientation is HORIZONTAL, meaning the cloth is lying flat, all of the y coordinates of the point masses are set to 1. Otherwise, if the cloth's orientation is VERTICAL, the z position of point masses is set to a small random offset (in the range -1/1000 to 1/1000). We then check if the point mass is pinned and pass in the appropriate boolean value when instantiating the PointMass objects.

Once we have generated all of the point masses, we create the springs to apply the structural, shear, and bending constraints, following these set of rules (directly brought from the project specs):

  1. Structural constraints exist between a point mass and the point mass to its left as well as the point mass above it.
  2. Shearing constraints exist between a point mass and the point mass to its diagonal upper left as well as the point mass to its diagonal upper right.
  3. Bending constraints exist between a point mass and the point mass two away to its left as well as the point mass two above it.

Part 3: Handling collisions with other objects

In this section, we implement how the cloth collides and interacts with other objects that are present in the scene. The main parts of our implementation are Sphere::collide, Plane::collide, and updating Cloth::simulate. For Sphere::collide, we check if a point mass is intersecting with or inside the sphere. If it is, we “bump” it to the surface of the sphere by computing the surface intersection point, where the point mass should have intersected the sphere. Once we have this intersection point, called the tangent point, we compute the correction vector. Now that we have the correction vector, we compute the point mass’s new position, which is its last position adjusted by the correction vector scaled down by friction. Our implementation for Plane::collide, is the analogous counterpart for a plane. The only differences are that we compute where the point mass should have intersected the plane, rather than a sphere, and that the tangent plane is computed using a small constant SURFACE_OFFSET to ensure that we’re on the side of the plane before the mass crossed the plane. Finally, as for Cloth::simulate, we simply added a loop that iterates through every PointMass object and tries to collide it with every CollisionObject.

Part 4: Handling self-collisions

  1. Show us at least 3 screenshots that document how your cloth falls and folds on itself, starting with an early, initial self-collision and ending with the cloth at a more restful state (even if it is still slightly bouncy on the ground).

In this section, we add self-collision to our implementation. Without self-collision, we can see that the cloth passes through itself when it falls and folds.

Example of cloth clipping through itself after it fell

After adding self-collision, the cloth can fold normally and stack onto itself. This is done by spatial hashing. First, we partition the 3D space into 3D boxes with dimensions w∗h∗t. We then take the coordinates of each point mass and snap it to the closest 3D box using floor. Given the coordinates of the 3D box containing the point mass, we create a unique hash using the 3D coordinates of the box. Referring to stackoverflow, we used the equation hash = (x * P1 + y) * P2 + z to find a unique hash where P1 and P2 are large prime numbers. This is used as the key to our hash table, which maps to a vector<PointMass *> containing all the point masses in the 3D box. Then, for every point mass, we look up all potential candidate points for collision by checking all points in the 3D box from the hash table. If there is a collision (points are less than 2 * thickness apart), we apply a correction vector such that the point mass is moved to a distance 2 * thickness away from the other point mass.

As we can see below, now when the cloth falls, if the point mass on the cloth hits another point mass on the cloth, they repel each other, as a collision force acts on that point. This is what causes the cloth to fold, such that no point on the cloth intersects itself, preventing clipping. Early on as the cloth falls, the cloth wrinkles before it piles up on itself in the initial self-collision. At mid self-collision, we can see that instead of the top sections of the cloth falling through the existing pile, it rolls out and unfolds. Finally, in the rest state, since we do not have damping factors like friction in this simulation, the cloth's point masses continue to repel each other and result in the cloth almost completely unfolding and flattening on the plane.

  1. Explain in your own words what a shader program is and how vertex and fragment shaders work together to create lighting and material effects.

For this part of the project, we implemented a couple of basic GLSL shader programs. They are isolated programs that run in parallel on the GPU, allowing for an acceleration of computation, that execute sections of the graphics pipeline and outputs a single 4 dimensional vector. Our GLSL shaders, which have functions similar to C, have two types: vertex shaders and fragment shaders.

  • vertex shaders: as the name suggests, vertex shaders generally apply transforms to vertices. They modify their geometric properties like position and normal vectors, and they write the final position of the vertex to gl_Position. They also store additional computations that’ll be used in the fragment shaders.
  • fragment shaders: as the name suggests once again, fragment shaders operate on fragments after rasterization and before computing the color for the framebuffer in the graphics pipeline. These shaders take in the fragments’ geometric attributes after rasterization, and they use this information to compute a color, which is written to out_color.

Overall, in our graphics pipeline, we have rasterization, then our vertex shader, then our fragment shader; the output of the vertex shader becomes the input of the fragment shader.

Textures can be used for more than just selecting colors for a mesh; using textures, we can also encode a height map that can later be applied in a shader program. In our implementation, this was accomplished through two different approaches: bump mapping and displacement mapping. Starting with bump mapping, we modify the normal vectors of an object so that the fragment shader gives the illusion of detail on an object, such as bumps. This modified normal vector was then directly fed into the implementation of Blinn-Phong that we had from earlier parts. To calculate the modified/displaced normal vector, our approach involved using a tangent-bitangent-normal (TBN) matrix to do the computations in object space for efficient calculations then map everything to model space at the end. The calculations we did consisted of dU and dV, which represents how the height changes when we make small changes in u or v.

From here, displacement mapping was more or less an extension of bump mapping. We do the same set of calculations as bump mapping, but we also do additional calculations that update the position of the points to take into consideration the height map as well as the new geometry.
